# Keep built-in Annotation recipes static and centralize localization

V1 adds first-party Annotation recipes as private, statically imported modules behind the sole `createScenePlan` Interface. Each exact recipe version owns one closed grammar, role vocabulary, semantic-ID policy, declared Semantic correction slots, and complete `en` and `zh-CN` Localization catalogs; a common finalizer owns source identity, recipe lookup, locale resolution, correction application, graph validation, limits, provenance, and diagnostics. The public author path remains the bare `recipe + source + locale`: the package lockfile pins the recipe Implementation and the Scene plan records its exact version, so V1 adds no author-facing version selector. There is no public registry, runtime catalog loader, dynamic discovery, dependency-injection port, or executable recipe DSL.

The Localization locale selects generated reader text, not the language of canonical author source. To preserve the closed Scene plan V1 shape and avoid negotiation drift, core compares the frozen supported catalog tags case-insensitively, materializes their canonical spelling, and otherwise fails: it performs no RFC 4647 fallback, aliasing, content detection, ambient-locale lookup, or CLDR maximization. Grammar ambiguity fails closed with a namespaced recipe diagnostic and every bounded candidate range rather than source-order or fuzzy guessing. Annotation renderers put real `lang` metadata on generated title and legend content while source inherits the host document language; CJK font choice and other Visual style remain theme concerns and never enter the Scene plan.
