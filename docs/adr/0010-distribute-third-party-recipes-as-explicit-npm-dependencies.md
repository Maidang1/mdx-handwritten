# Distribute third-party Annotation recipes as explicit npm dependencies

Third-party Annotation recipes are distributed only as standard ESM npm packages and become available only when a host explicitly installs, imports, and supplies them to a new Configured Scene compiler. The existing root `createScenePlan` Interface and first-party static table remain unchanged; there is no package scanning, remote catalog, source-driven dynamic import, ambient mutable registry, or renderer-side loading. Importing a Recipe package grants its JavaScript the same build-time trust as any other npm dependency and is not a sandbox claim, while the Scene Module still protects its output boundary by accepting only bounded semantic drafts and owning normalization, localization, Semantic corrections, validation, limits, provenance, diagnostics, and final Scene plan construction. npm SemVer, the Recipe package protocol, exact Annotation recipe versions, Localization catalog versions, and Scene plan schema versions remain independent compatibility axes, and any unknown, duplicate, incompatible, throwing, or invalid recipe fails closed without fallback or implicit migration.

This decision extends ADR 0003's single root Interface with one recipe-specific Configured Scene compiler capability and supersedes only ADR 0006's rejection of a public recipe construction port. It retains ADR 0005's rejection of a generic recipe registry and ADR 0006's private static ownership of built-ins: Configured Scene compilers always contain the installed first-party recipes, copy and freeze third-party configuration metadata and function references at construction, and never mutate the root Interface. A function's closure and module state remain trusted package behavior rather than frozen or sandboxed state.
